
\section{RQ2: In what ways are LLMs employed to secure SSC?}
\label{sec:rq2}

\par This Research Question examines how LLM defend Software Supply Chain
security, by means of reviewing collected literature. We begin by 
introducing the major attack vectors of SSC breaches, and then review
how LLMs are leveraged to mitigate these kinds of attacks.

categorized the three major attack vectors as follows\citep{sscdirection}:

\subsection{Mitigation of Malware and Vulnerabilities}\label{subsec:vec1}

\subsubsection{Which components of Software Supply Chain are secured by LLM?}
We made a qualitative analysis of components whose security is protected
by LLM-based methodologies. We classify components of Software Supply Chain
by the type of hosted artifacts, as follows:

\begin{itemize}
  \item \textit{Package Manager} usually hosts artifacts compiled from the
  source code which can be directly imported as dependencies. Examples
  of this include \texttt{npm}, \texttt{maven}, and \texttt{PyPI}
  \footnote{These sites provide both the source code and compiled byte-code.}.

  \item \textit{Code-hosting Platforms} such as GitHub provides the source code
  of repositories as-is. A \textit{repository} is a collection of
  source code files which is often embedded as a dependency.

  \item \textit{AI models} include both providers of pre-trained weights
  and their applications (e.g. LLM-integrated Application (LIA)).

  \item \textit{LLM Harness} refers to utilities of the Large Language Model
  enabling it to autonomously run tasks. Providers of MCP server,
  skills and plugins of LLM-based agents fall in this category.
\end{itemize}

pertain the majority of "LLM for SSC Security" studies, recent years

\paragraph*{Threat Detection \& Analysis}
The goal of threat detection and analysis is to identify potential 
vulnerabilities (e.g. \cite{10.1007/s10664-026-10832-4,10.1016/j.jss.2024.112234,10.1109/ICSE48619.2023.00089})
and  malicious behaviors in software package (e.g. \cite{10.1145/3577923.3583657,10.1145/3705304,11029752}) 
before they can be triggered by the attacker (i.e. vulnerabilities and malicious behaviors
are the major source of SSC threats).
The detection of either vulnerabilities or malicious behaviors can be
modeled as a binary classification problem -- classifying a code snippet
as benign or vulnerable/malicious.

\par Vulnerabilities are seemingly benign defects in the source code which usually does not have
tangible consequences but can lead to system failures or theft of sensitive information.
The use of LLM in vulnerability detection is largely motivated by their capabilities of understanding
code semantics and contextual reasoning. With some LLM-enhancement techniques (Section~\ref{sec:rq1}),
these language models have shown promising results in  vulnerability detection.
For instance, H. Shu et al. leveraged a combination of  few-shot prompting 
and fine-tuning to improve the performance of LLM
in multi-lingual vulnerability detection \citep{10.1007/s10664-026-10832-4}; 
Y. Yang employed fine-tuning and chain-of-thought prompting for exceptional
detection outcome \citep{10.1016/j.jss.2024.112234}.

\par LLMs have demonstrated similar potential for detecting the malicious behaviors
in software packages. Software backdoors tend to undermine authorization mechanisms,
whereas ransomwares render the system unusable. Most malware detection studies
(including \cite{10.1145/3705304, 10695665, 11029752})
follow a "LLM as a static analyzer" fashion to overcome the low generalizability
of static analysis tools, where LLM search the code repository
for clues, given a set of pre-defined knowledge and patterns of malicious behaviors.
On the contrary, T. Ganz et al. developed a novel method where an LLM is
employed to vet the commit history reflecting development process for clues
of malicious code injection\citep{10.1145/3577923.3583657}.

\paragraph*{Security Testing}
While the primary purpose of software testing is to ensure functional correctness,
testing techniques can also uncover previously unknown vulnerabilities.
Large Language Models have been adopted to enhance \textit{mutation testing} 
\citep{10.1145/3756681.3756991}, a technique to assess and enhance the
effectiveness of test suite. Also in \textit{fuzz testing} whose goal
is to trigger software crash by generating or mutating test inputs,
LLM have been employed to directly generate a set of diverse test inputs
\citep{fuzz4all}. 

\paragraph*{Reverse Engineering}
Some software packages undergo a series obfuscations before they are released
by a \textit{package manager}, in order to conceal the source code from which
they are compiled. These obfuscations posit extra complexities for
\textit{threat detection \& analysis}. Such challenge can be resolved
by \textit{binary code analysis} -- directly reasoning the binary byte code
for software vulnerability of malicious behaviors \citep{11519107}, 
or \textit{reverse engineering},
which tries to reconstruct the source code from the compiled byte code
\citep{YU2026104923}. For instance, S. Meng et al. incorporates Large Language Model
and knowledge graph for hallucination-free function-granularity malicious
code detection \citep{11519107}.

\paragraph*{Red Teaming} By mocking the intrusion behavior of the attacker,
\textit{Red Teaming} is a technique for unveiling the software vulnerabilities
in production environment. X. Fan et al. proposed an LLM-assisted Red Team
framework for dynamic Cyber Kill Chain simulation which demonstrated
a 99.2\% accuracy in vulnerability detection \citep{10.1145/3803633.3803641}.

\paragraph*{Threat Modeling}
The goal of threat modeling is to "create an abstraction of the system; profiles 
of potential attackers, including their goals and methods; and a catalog of potential
threats that may arise" \citep{threatmodeling}.
\textit{Threat modeling} is integral to firstly understanding potential system
threats and ultimately threat mitigation.
 Recent years have seen attempts
to automate threat modeling with Large Language Model across several
disciplines, including hardware design \citep{11022932}, pre-trained AI models
\citep{10764813}, LLM-Integrated software applications \citep{10.1145/3756681.3757083},
and MCP \citep{11406667}.

\paragraph*{Threat Validation} While \textit{threat detection \& analysis}
help elicit software security threats, extra efforts is needed to verify
their existence and possibility of exploitation, hence \textit{threat validation}.
W. Mbaka and K. Tuma conducted an empirical study on the usefulness
of LLM-assisted \textit{threat validation} and concluded that practitioners
benefited from LLMs despite their not always providing conclusive results
\citep{10.1007/s10664-026-10837-z}. Z. Jiang et al. proposed an innovative
agentic framework while fully utilizing LLM's reasoning capability
to identify vulnerability-revealing issue reports \citep{10.1145/3801149}.
C. Ma et al. leveraged LLM to help structure Cyber Threat Intelligence (CTI)
reports and to automatically analyze possible attack techniques
\citep{MA2025104141}. By integrating LLM-enhancement techniques (Section~\ref{sec:rq1})
and threat knowledge, LLMs have demonstrated potential of automating
the formerly manual work of \textit{threat validation}.

\paragraph*{Affected Library Detection}
Given a set of reported vulnerabilities, \textit{affected library detection}
aims to help practitioners determine whether their software is affected by
the vulnerabilities in the dependent libraries and take actions if affected. 
J. Xu et al. adopted
parameter-efficient fine-tuning \citep{hu2022lora} and a customized
hallucination elimination algorithm for classifying affected libraries from
16238 NVD (National Vulnerability Database) reports \citep{10.1007/s10515-025-00540-6}.
Also, VULLOC, a framework integrating knowledge augmentation 
and fine-tuning \citep{11495563},  was proposed by  J. Xu et al.  for mapping
NVD reports to localizing affected libraries and versions.

\par Code hosting platforms such as \texttt{GitHub} allow the user to
define a set of \textit{workflows} which are automatically executed
when the source code changes. These \textit{workflows} perform code linting,
software testing, as well as building artifact. Vulnerabilities
in the \textit{workflows} may allow attackers to steal sensitive 
information or inject malicious code.  To this end, Benedetti
et al. proposed an automated approach to assess the safety of
\textit{GitHub Actions} \citep{10.1145/3560835.3564554}, resulting
in a total of 24,905 security issues across 50 code repositories.

\par Certain Software Supply Chain attacks rely on the weakness of
build system (e.g. that resides in a package manager) to
inject malicious code. To alleviate these attacks, Hassanshahi et al.
built a tool to detect the presence of logical discrepancies between
the source code and built artifact, a step towards transparency and
trustworthiness of software packaging process \citep{10.1145/3605770.3625213}.

\subsection{RQ2 -- Conclusion}\label{subsec:rq2end}

This section delves into the attack vectors of Software Supply Chain attacks
and mitigation techniques for attack vector 1 and 2 (in section~\ref{subsec:vec1}
and section~\ref{subsec:vec2}, respectively).
For attack vector 1, we contributed a comprehensive taxonomy of mitigation
techniques, as shown in Figure~\ref{fig:vec1taxo}. Moreover,
we briefly categorized potential threats to build system and infrastructure,
in Figure~\ref{fig:vec2ov}. We conclude that with domain-specific
knowledge engineering and LLM-enhancement techniques discuss in RQ1,
LLMs are promising in securing Software Supply Chain.

\paragraph*{Research Gaps and Directions}

\begin{itemize}
  \item We have observed a scarcity of attack vector 2 and 3 mitigation
  methods, as shown in Figure~\ref{fig:rq2dist}. This calls for future research
  into Social Engineering detection, LLM-assisted analysis of workflows,
  etc.

  \item There are also concerns of LLM's generalizability,
  consistency, and reproducibility when applied to SSC security.
  Due to their probabilistic nature, the intrinsic hallucination
  of LLMs can lead to failed threat detection, erroneous threat analysis,
  and most importantly, inconsistent results.
  Furthermore, we observed a tendency to fine-tune the LLM before
  application to down-stream tasks. We argue that the generalizability
  (to unseen input) of fine-tuned LLM in SSC security must be experimented before they
  are put into production.

  \item Recent advancements in agentic framework bring promise to
  automated defense of SSC security. Compared to fine-tuning, 
  incorporating LLM-based agents and domain knowledge can eliminate
  the need to collect dataset (required for tuning) and the need of
  computation power.
\end{itemize}
