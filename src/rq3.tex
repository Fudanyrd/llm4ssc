\section{RQ3 -- Potential SSC Risks Imposed by LLM}
\label{sec:rq3}

\par As shown in Figure~\ref{fig:llmrisk},
 we summarized possible vulnerabilities  of an LLM, which
 may posit attack surfaces to the Software Supply Chain,
 across its lifecycle --
\textit{training \& tuning} stage, \textit{inference} stage,
and \textit{agent} stage.  This section then
delves into how these vulnerabilities turn against
Software Supply Chain security.

\subsection{Implications of Training Stage Vulnerabilities}
\label{subsec:imptrain}
In its training and fine-tuning process, Large Language Models
are susceptible to \textit{data poisoning} and \textit{backdoor injection}
attacks. Speaking of Code Language Models (CLMs), its training data
may be contaminated by malicious or vulnerable code snippets.
Consequently, in their inference stage, the generated code contains
similar patterns of vulnerability or malice; certain LLM-based
bug detector may suffer reduced effectiveness. As an increasing proportion
of code is written by CLMs, \textit{data poisoning} is capable of
polluting Software Supply Chain with problematic software packages.
For instance, M. Jahanshahi et al. made a large scale investigation
into the use of Open Source Software in LLM training, and observed
a drop in vulnerability rate when the LLM is trained with cleaner
code (e.g. whose CVEs are resolved) \citep{11028470}. Such observations
calls for data poisoning mitigation techniques (e.g. \citep{11344010}) to 
sanitize the defects in the training data.

% SSC attack vector 1.
\par Researchers have noticed the existence of \textit{backdoor}s in
these CLMs, which enabled the attacker to manipulate a model's response
with a trigger in the prompt to obtain malicious code from the model's
output \citep{li2021backdoorattackspretrainedmodels}. 
Y. Qu et al. conducted a review of LLM backdoor attacks, 
fine-tuning based  methods, and
applications (including deliberate malicious code injection)
\citep{10.1016/j.infsof.2025.107707}. Like \textit{data poisoning},
\textit{backdoor attack} can also inject malicious package into
Software Supply Chain.


\subsection{Implications of Inference Stage Vulnerabilities}
\label{subsec:impinfer}

\paragraph*{Jail Breaking} This kind of attack is employed to bypass the ethical
alignment\citep{carlini23aligned} and security restrictions of an LLM, so that malignant 
can be generated \citep{10.1145/3716628}. In the scenario of Software Supply Chain, attackers
can adopt this attack for malicious code generation and
vulnerability injection \citep{10.1007/s10515-026-00628-7}.

\paragraph*{Adversarial Attack}  After a CLM is published,
 adversarial samples  can be generated to perturb its prediction
process and yield unexpected output. It is assumed that 
adversarial attacks on code models can introduce vulnerabilities and
backdoors in the software under development \citep{10.1007/s10515-024-00464-7},
but this attack surface is yet to be explored by future research.

\paragraph*{(Unintended) Vulnerable Code Generation}
Code models have been extensively used in programming tasks.
Meanwhile, their intrinsic hallucinations have given rise to vulnerabilities
in generated code. For instance, experiments conducted by H. Pearce et al. 
showed that around 40\% of 1689 \textit{GitHub Copilot} \citep{copilot}
generated programs are vulnerable (i.e. 
CWEs are detected by static analyzer) \citep{copilotsecurity}.
S. Ullah et al. made an extensive evaluation of several LLMs' ability
to detect security vulnerabilities in source code and found a limited
capability in reasoning about  vulnerability \citep{ullah2024llms}.
These results suggest that LLM-generated code should be vetted
for potential vulnerabilities before deployment.

\paragraph*{Guided Malware Generation}
Equipped with \textit{adversarial attack} and \textit{jailbreak attach}
methods, attackers can leverage LLM to generate and inject malicious
code into components of Software Supply Chain. For instance,
E. Iturbe et al. explored LLM's capabilities in crafting attack scripts
, resulting in a success rate of 16\% \citep{ITURBE2024104077}. 
M. Fard et al. leveraged an LLM to inject software backdoor,
which can bypass the audit of developers with high efficacy \citep{llmbackdoordet}.


\paragraph*{MLSys Breach}
Most LLM are deployed on a 
Machine Learning Systems (MLSys), or an LLM-integrated applications.
Breaching these infrastructures is likely to evade the prediction
of an LLM. 

\subsection{Implications of Agent Stage Vulnerabilities}
\label{subsec:impagent}

\par Equipped with agent harness (including tools, memory, etc.),
an LLM is turned into an \textit{agent} capable of autonomously 
completing tasks. This section discusses the implications of
vulnerabilities in the agent harness. It is worth noting that 
\textit{social engineering attack}
and \textit{penetration testing} are remotely associated with Software
Supply Chain, but we kept them in this section because they are
part of our search results (Section~\ref{subsec:search}). 

\paragraph*{Malicious Skill or Plugin} 
The skill or plugin (discussed in section~\ref{sec:rq1}) of a coding agent
may contain covert prompts guiding the LLM to inject vulnerable
or malicious code. Though no skill-guided SSC attack has been published,
the presence of malicious skill has more severe consequences, including
credential theft, data loss and system control \citep{skillvulns}.

\paragraph*{Compromising MCP \& Agent Framework}
MCP server is the bridge between an agent and its data source (details
are in Section~\ref{sec:rq1}). The risks of MCP servers include remote
code execution, tool poisoning, and \textit{prompt injection}
\citep{10.1145/3814959,mcpvulns}. Attackers may employ a simple
\textit{prompt injection} to embed a vulnerability or backdoor
in the code agent's workspace, thereby compromising a software package.
Likewise, the vulnerability in a \textit{agent framework}
\footnote{A software that manages the harness of an LLM-based agent, e.g. \texttt{langchain}.}
has similar implications.

\paragraph*{Penetration Testing}
LLM-based agents may be employed to perform automated penetration testing
and discover previously unknown vulnerabilities
 \citep{10970188,11418046,10.1145/3766895}. This may enable the attacker
to gain an upper hand, especially when the vulnerability is not
disclosed.

\paragraph*{Social Engineering Attack}
LLMs can be leveraged to produce realistic contents for successful
social engineering attacks. For instance, M. Chowdhury 
et al. examined the use of ChatGPT in developing trojans,
rootkits, and phishing tools \citep{10187385}.
These attacks  can lead to malware distribution or credential theft.

\subsection{RQ3 -- Summary}\label{subsec:rq3end}
\par As shown in Figure~\ref{fig:llmrisk}, we made a comprehensive study
of leveraging LLM's intrinsic vulnerabilities for attacking on Software Supply
Chain. Section~\ref{subsec:imptrain}, \ref{subsec:impinfer}, and \ref{subsec:impagent}
discussed potential SSC attack techniques during different stages of LLM's lifetime.
Apart from these, LLM may be employed by attackers for \textit{reverse engineering},
\textit{penetration testing}, \textit{forging bug report}, 
and \textit{malicious code obfuscation},
threatening the safety of SSC.

\paragraph*{Exploitation of SSC Attack Vectors 
\footnote{Attack vectors are discussed in Section~\ref{sssec:avectors}.}} 
Because of LLM's tendency to
generate vulnerable code \citep{copilotsecurity} and susceptibility
to a variety of attacks (such as backdoor, adversarial sample, etc.), 
the exploitation of attack vector 1  
will probably become predominant -- by unintentionally or deliberately
generate malicious or vulnerable code and publish it on the web.
We have also seen exploitation of attack vector 3 -- through \textit{jailbreaking} the
LLM and prompt it to aid \textit{social engineering} \citep{10187385}.
Nevertheless, exploitation of attack vector 2 (the build system) is
currently rare.

\paragraph*{Future Research Directions} 
\begin{itemize}
  \item The exploitation of attack vector 2 (targeting the build system)
  is still lacking, which calls for future research. Possible methods
  include malicious script injection into \textit{workflow} file,
  automated malicious code injection during the build process,
  recognition of hard-coded tokens or credentials.

  \item Because of tremendous interest in LLM-based vulnerability
  detection, malware detection, and security testing methods (discussed 
in Section~\ref{subsec:vec1}), it is still unclear whether
existing attach techniques can bypass these checks.
 Future work is needed to evaluate the effectiveness of attack vector 1
 in the presence of state-of-the-art threat detectors.
\end{itemize}
